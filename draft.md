# [KUS-TUDY] 우수그룹 자기추천서 — Big O(S)

> 제출용 PPTX 제작을 위한 기반 자료. Big O(S) 팀의 1~12주차 활동(알고리즘·운영체제 + LLM 프로젝트)과 결과물 레포지토리를 근거로 정리하였다.

---

## 1. 팀소개

### 팀명
    Big O(S) - 알고리즘과 운영체제 수업을 같이 공부한다는 뜻으로 두 과목의 용어를 혼재하였다.

### 참여계기 및 동기
    우리 4명은 모두 군휴학 후 올해 복학하였다. 2년 전만 해도 AI는 커닝을 위한 도구에 불과했지만, 2년이 지난 지금 AI는 개발자에게 없어서는 안 될 생산 도구가 되었다. 뒤처진 기술 트렌드를 따라잡기 위해, AI를 적극 활용하는 알고리즘·운영체제 수업을 통해 컴퓨터 과학 전공 지식과 기술 활용 능력을 함양한다.

### 팀 구성
    컴퓨터융합소프트웨어학과 4인으로 구성된 팀이다. 전공 기초 과목(알고리즘·운영체제) 학습을 기반으로,
    9주차부터는 LLM(Upstage Solar)을 결합한 팀 프로젝트로 확장하였다. 팀 내 역할은
    **Frontend / Backend / LLM·Prompt / PM(일정·문서)** 4개 축으로 나누어 네 명이 각자 주된 역할을 맡고,
    필요에 따라 복수 역할을 겸하는 방식으로 운영하였다. (총 4인)

    2. 팀원: 이태진 / 3학년 / 2022271344 / 과학기술대학 / 컴퓨터융합소프트웨어학과

## 2. 학습목표

### 전체 팀 목표
    매주 강의에서 배운 알고리즘·운영체제 핵심 개념을 곧바로 동작하는 코드·웹앱·시스템으로 구현하여, 개념에 대한 깊은 이해와 실전 응용 능력을 동시에 기른다. 학기 후반에는 LLM(Upstage Solar)을 결합한 팀 프로젝트로 학습 성과를 집약해 실제 결과물을 도출하고, LLM 기반 개발 방식에 적응하는 것을 전체 목표로 삼는다. 이 과정에서 '이론 → 구현 → 검증'의 사이클을 팀의 표준 학습법으로 정착시킨다.

### 개인별 목표
    손한주 - 향후 LLM의 하네스 엔지니어링 능력을 향상해 개발능력의 기반을 다진다.
    이태진 - 운영체제·시스템 프로그래밍의 저수준 동작 원리를 깊이 이해하고, AI 협업 도구를 활용해 커널 수준의 코드까지 직접 다루는 시스템 개발 역량을 기른다.
    송명준 - 효율적인 AI의 사용법을 숙달해 개발의 품질을 향상시킨다.
    박상언 - 추천 시스템, AI 서비스 백엔드, LLM 기반 애플리케이션 개발자를 위한 개발능력의 기초를 다진다.

### STUDY 영역
    전공 교과 영역 : **알고리즘 + 운영체제 (CS 전공 기초)**
    - 알고리즘: 분할 정복, 동적 계획법, 탐욕법, 정렬/탐색, 탐색 트리(BST/RBT/B-트리), 해싱, 그래프
    - 운영체제: 프로세스/스레드, CPU 스케줄링, 동기화, 대용량 저장장치, 그리고 xv6 커널 직접 수정
    - 응용 영역: LLM(Upstage Solar Pro)·AI 협업 도구(Claude Code, NVIDIA NIM)를 결합한 웹·시스템 프로젝트
    - 학습 교재: 『Introduction to Algorithms』(Thomas H. Cormen 외, 알고리즘) · 『Operating System Concepts』(Abraham Silberschatz 외, 운영체제)

## 4. 학습목표를 위한 노력, 전략

### 스터디 준비 / 진행 / 개선을 위한 노력
    - **고정 루틴**: 매주 오프라인으로 모여(활동 일시 기록: 3월말~6월, 회당 약 2~5시간) 강의 진도에 맞춘 개념
      정리 → 개념을 적용한 코드/웹앱 구현 → 실행·검증 → 퀴즈/시험 대비의 순서로 진행하였다.
    - **이론과 구현의 연결**: 매 주차 강의에서 배운 개념을 반드시 "동작하는 산출물"로 만들었다(1주차 Closest Pair
      웹앱, 2주차 병렬 히스토그램, 3주차 표절검사기, 4주차 스케줄러 시뮬레이터 …).
    - **개선**: 시험 대비 주차(5주차)에는 오픈북 시험 특성에 맞춰 교과서 정독·인덱싱과 NotebookLM 예상질문 방식으로
      전략을 바꾸는 등, 매주 활동 방식을 목표에 맞게 조정하였다.

### 팀별 역할 / 기준 / 과제 / 규칙
    - **역할 분배**: Frontend / Backend / LLM·Prompt / PM(일정·문서) 4개 축으로 분담(9주차 정립).
    - **결과물 기준**: 모든 주차 활동은 `weekNN/weekNN.md` 보고서 + 실행 가능한 코드/웹앱으로 남기는 것을 원칙으로 함.
    - **버전 관리 규칙**: 결과물을 팀 GitHub 레포지토리로 관리하고 주차별로 커밋하여 기록을 누적.
    - **퀴즈·시험 대비 규칙**: 매주 퀴즈 오답을 공유하고 다음 주 출제 경향을 함께 예측·대비(2주차 퀴즈 전원 만점).

## 5. 활동 소감

### 스터디를 진행하며 잘한 점 / 아쉬운 점
    **잘한 점**
    - 강의에서 배운 개념을 매주 빠짐없이 동작하는 코드/시스템으로 옮겨, 이론과 구현의 간극을 좁혔다.
    - 단순 알고리즘 문제 풀이를 넘어, 학기 말에는 xv6 커널을 직접 수정해 시스템 콜·스케줄러·동기화를 구현하는
      수준까지 난이도를 끌어올렸다.
    - 모든 활동을 보고서와 GitHub 결과물로 남겨 학습 과정을 누적·추적할 수 있게 하였다.
    - 2년의 공백(군휴학) 동안 달라진 AI 개발 환경을, AI를 단순 보조가 아닌 실질적 생산 도구(LLM 하네스
      엔지니어링·AI 페어 프로그래밍)로 활용하는 방식으로 빠르게 따라잡았다.

    **아쉬운 점**
    - Liberal_OS에서 proxy_lock 직렬화로 인해 여러 LLM 에이전트의 호스트 호출을 진정한 병렬로 돌리지 못했고,
      live 모드 자동 검증·순차 baseline·결함 격리 테스트가 일부 보류되는 등 보완할 한계가 남았다.
    - 일부 주차는 시험·과제 일정과 겹쳐 결과물 완성도에 편차가 있었다.

### 스터디를 진행하며 얻은 결과·성과
    - 알고리즘·운영체제 전공 개념을 실제 구현으로 체화하고, 퀴즈 전원 만점·중간고사 대비 등 학업 성과로 연결.
    - LLM을 결합한 두 개의 완성 프로젝트(Liberal_algorithm, Liberal_OS)를 산출물로 확보.
    - "이론→구현→검증" 사이클과 AI 협업 워크플로우를 팀의 표준 학습법으로 정착.

### 팀원별 활동 소감
    **손한주 (대표 · LLM 하네스)**
    대표로서 매주 일정과 역할을 조율하면서, 특히 12주차 Liberal_OS에서 AI 코딩 에이전트가 xv6 커널 개발을
    자율적으로 수행하도록 만드는 하네스(자동 테스트·회귀 게이트·작업 루프)를 설계한 경험이 가장 값졌다.
    AI를 단순 질의 도구가 아니라 개발 사이클을 굴리는 주체로 다루며 '하네스 엔지니어링'의 중요성을 체감했고,
    동시에 proxy_lock 직렬화나 live 모드 자동화처럼 끝까지 다듬지 못한 한계를 정직하게 기록하는 일도 실력의
    일부임을 배웠다.

    **이태진 (운영체제·시스템)**
    평소 추상적으로만 알던 운영체제 개념이 실제 코드에서 어떻게 구현되는지를 xv6 소스(proc.c, scheduler,
    syscall)를 직접 수정하며 확인한 점이 인상 깊었다. 시스템 콜을 추가하고 우선순위 스케줄러와 메모리 쿼터를
    손보며 저수준 디버깅의 어려움을 체감했고, 이론과 구현 사이의 간극을 메우려면 더 깊은 시스템 이해가
    필요하다는 것을 절실히 느꼈다.

    **송명준 (AI 활용·품질)**
    Claude Code와 Upstage Solar 같은 AI 도구를 코드 리뷰·복잡도 최적화·문서화에 활용하면서, 같은 도구라도
    어떻게 묻고 어떻게 검증하느냐에 따라 결과 품질이 크게 달라진다는 것을 배웠다. AI가 제안한 코드를 그대로
    신뢰하지 않고 테스트와 벤치마크로 검증하는 습관을 들인 점이 잘한 부분이며, 효율과 정확성을 동시에 잡는
    AI 활용법을 더 다듬고 싶다.

    **박상언 (추천·AI 백엔드)**
    11주차 Liberal_algorithm에서 Union-Find 군집화와 Kruskal 최대신장포레스트로 취향 그래프를 만들고 그 위에
    LLM 추천을 얹는 백엔드 파이프라인을 설계하며, 결정론적 수치 계산과 LLM 해석의 역할을 분리하는 설계 원칙을
    체득했다. 추천 시스템과 AI 서비스 백엔드의 전체 흐름을 직접 구현해 본 점이 값졌고, 추천 결과를 정량적으로
    평가하는 부분은 더 보완이 필요하다고 느꼈다.

### 그 외 소감 및 활동 후기
    - 서로 다른 관심사(하네스·시스템·AI 활용·백엔드)를 가진 네 명이 한 주제를 각자의 시각에서 파고든 덕분에,
      혼자서는 닿기 어려운 범위(웹앱부터 커널까지)를 한 학기에 함께 경험할 수 있었다.

      각 주차 결과물을 시험 관점에서 재정리한다.

### 스터디를 통해 세운 앞으로의 계획
    - 최종 프로젝트 고도화: Liberal_OS의 향후 과제(순차 baseline 벤치, 결함 격리 테스트, 실제 virtio-serial
      드라이버, 우선순위 라운드로빈/에이징, 실 LLM 병렬 호출)를 다음 학기 연계 주제로 발전.
    - 전공 심화(시스템 프로그래밍/분산 시스템/AI 시스템)로 학습 범위를 확장하고, 스터디 운영 방식을 다음 학기에도 지속.
    - 각자의 개인 목표(LLM 하네스 엔지니어링 · AI 서비스 백엔드 · 시스템 개발 · 효율적 AI 활용)에 맞춰
      AI 기반 개발 역량을 지속적으로 심화한다.

### 팀원별 향후 계획
    **손한주** — LLM 하네스·에이전트 오케스트레이션을 더 깊이 공부해 Liberal_OS의 live 모드 안정화와 에이전트
    평가 자동화를 이어가고, AI가 주도하는 개발 워크플로우 설계 역량을 개발자로서의 기반으로 다질 계획이다.

    **이태진** — 시스템 프로그래밍·운영체제 후속 학습을 이어가 virtio-serial 드라이버 같은 커널 확장 과제에
    도전하고, 저수준 코드를 직접 다루는 시스템 개발자로 성장하기 위한 기초를 다질 계획이다.

    **송명준** — AI 협업 워크플로우를 팀 표준으로 정리하고, 테스트·검증 자동화와 코드 품질 도구를 익혀 AI를
    효율적이면서도 신뢰할 수 있게 쓰는 개발 습관을 굳힐 계획이다.

    **박상언** — 추천 시스템과 AI 서비스 백엔드를 심화 학습해 Liberal_algorithm을 API·데이터베이스를 갖춘 실제
    서비스 형태로 확장하고, LLM 기반 애플리케이션 개발자로 나아갈 계획이다.
