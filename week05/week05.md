    (1) 알고리즘 분석의 기초

    알고리즘 속성
    입력크기 n에 따른 시간복잡도 분석 지표
    (2) 주요 정렬 알고리즘 비교

    삽입 정렬(Insertion Sort): 거의 정렬된 데이터에 효율적이며, A[1…i−1]이 정렬되어 있다는 루프 불변성을 가집니다.
    힙 정렬(Heapsort): 완전 이진 트리인 힙 구조를 사용하여 최악의 경우에도 **O(nlogn)**을 보장하며 추가 메모리가 거의 들지 않습니다.
    병합 정렬(Merge Sort): 분할 정복 방식의 대표 예시로, 병합(Combine) 단계에 대부분의 연산이 집중됩니다.
    퀵 정렬(Quicksort): 평균 성능이 매우 우수하지만 피벗 선택에 따라 최악의 성능(O(n2))이 나올 수 있어 랜덤 피벗이 권장됩니다.
    (3) 알고리즘 설계 패러다임

    분할 정복(Divide and Conquer): 문제를 독립적인 하위 문제로 나누어 해결합니다 (예: 병합 정렬, 이진 탐색).
    탐욕 알고리즘(Greedy): 매 순간 최선의 선택을 하며, 탐욕적 선택 속성과 최적 부분 구조가 증명되어야 사용 가능합니다.
    동적 계획법(Dynamic Programming): 중복되는 하위 문제가 있을 때 결과를 테이블에 저장(Memoization/Tabulation)하여 지수 시간을 다항 시간으로 단축합니다.
    (4) 주요 DP 문제 점화식

    LCS(최장 공통 부분 수열): 문자가 같으면 대각선 위+1, 다르면 위쪽과 왼쪽 중 큰 값을 취합니다.
    행렬 경로 문제: 현재 셀 값에 위쪽이나 왼쪽 셀 중 더 큰 누적합을 더해갑니다.
    피보나치 수열: 단순 재귀(O(2n))를 DP로 해결하면 **O(n)**에 해결 가능합니다.
    

    

    2. 운영체제

    Chapter 1 & 2: 운영체제 서론 및 구조

    운영체제의 정의: 하드웨어를 관리하고 응용 프로그램에 환경을 제공하는 '정부'와 같은 프로그램(커널)입니다
    보호 메커니즘: **이중 모드 운영(User/Kernel mode)**을 통해 사용자 프로그램이 하드웨어에 직접 접근하는 것을 차단하며, 서비스 요청은 시스템 콜을 통해서만 가능합니다
    구조적 유형: 모든 기능을 하나에 담은 단일 구조, 최소한의 기능만 커널에 둔 마이크로커널, 그리고 현대의 혼합 구조 등이 있습니다
    Chapter 3: 프로세스(Process)

    개념: 실행 중인 프로그램으로, 텍스트, 데이터, 힙, 스택 섹션으로 구성된 메모리 레이아웃을 가집니다
    상태: New, Ready, Running, Waiting, Terminated 상태를 오가며 스케줄링됩니다
    생성: fork()로 복제하고 exec()로 새로운 프로그램을 로드합니다
    IPC: 협력적 프로세스 간 통신을 위해 공유 메모리(빠름, 동기화 필요)와 메시지 전달(안전, 분산 시스템 유리) 모델이 사용됩니다
    Chapter 4: 스레드(Thread)와 동시성

    특징: 프로세스 내 CPU 이용의 기본 단위로, 코드와 데이터를 공유하면서 독립적인 PC와 스택을 가집니다
    병렬성 vs 동시성: 여러 작업을 동시에 진행하는 것과 실제로 동시에 실행하는 것의 차이를 이해하고, 암달의 법칙을 통해 병렬화의 한계를 학습했습니다
    암시적 스레딩: 스레드 풀, Fork-Join, OpenMP 등을 통해 스레드 관리를 라이브러리에 위임하는 방식을 배웠습니다
    Chapter 5: CPU 스케줄링

    알고리즘: FCFS(단순, 호위 효과), SJF(최적 평균 대기 시간), RR(시간 할당량 기반 공평성), Priority(우선순위, 기아 문제) 등을 비교했습니다
    현대적 스레드 스케줄링: Linux의 **CFS(Completely Fair Scheduler)**는 vruntime과 레드-블랙 트리를 사용하여 공정한 CPU 할당을 보장합니다

    Q: fork()와 vfork()의 차이는?
    A: fork()는 주소 공간을 복사(COW 방식 사용)하지만, vfork()는 부모의 주소 공간을 직접 공유하며 부모를 일시 중지시킵니다. 주로 바로 exec()를 호출할 때 성능을 위해 사용합니다
    Q: 스케줄링에서 '기아(Starvation)' 현상이란?
    A: 우선순위가 낮은 프로세스가 더 높은 순위의 프로세스들에 밀려 영원히 CPU를 할당받지 못하는 상태입니다. 이를 위해 대기 시간에 비례해 우선순위를 높여주는 노화(Aging) 기법을 사용합니다
    [활동 3] NotebookLM 기반 예상 질문 도출 및 답변

    예상 질문 1: 커널 모드와 사용자 모드를 구분하는 기준과 그 목적은 무엇인가?
    답변: 하드웨어의 **모드 비트(0은 커널, 1은 사용자)**를 통해 구분합니다. 목적은 사용자 프로그램이 불법적인 메모리 접근이나 하드웨어 제어를 시도하여 시스템을 망가뜨리는 것을 방지하고 운영체제를 보호하기 위함입니다
    예상 질문 2: 암달의 법칙에 따르면 코어 수를 늘려도 성능 향상이 제한되는 이유는?
    답변: 프로그램 내부에 병렬화할 수 없는 **순차적인 부분(Serial fraction)**이 존재하기 때문입니다. 코어 수(N)가 무한해도 가속도는 1/S를 넘을 수 없으므로, 코어 추가보다 순차 실행 비중을 줄이는 것이 중요합니다
    예상 질문 3: Linux에서 스레드와 프로세스를 다루는 독특한 방식은 무엇인가?
    답변: Linux는 둘을 엄격히 구분하지 않고 모두 **'태스크(Task)'**라고 부릅니다. clone() 시스템 콜 호출 시 전달되는 플래그(CLONE_VM, CLONE_FILES 등)에 따라 자원 공유 수준을 결정하여 스레드나 프로세스처럼 동작하게 합니다
    예상 질문 4: FCFS 스케줄링의 '호위 효과(Convoy Effect)'를 설명하고 이를 해결할 알고리즘은?
    답변: 긴 CPU 버스트를 가진 프로세스가 CPU를 점유하여 뒤의 짧은 프로세스들이 오래 기다리게 되는 현상입니다. RR(Round-Robin) 알고리즘은 시간 할당량을 두어 CPU를 강제로 뺏음으로써 이 문제를 완화하고 응답 시간을 개선합니다
    예상 질문 5: 생산자-소비자 문제에서 '유한 버퍼(Bounded Buffer)' 사용 시 발생하는 동기화 이슈는?
    답변: 버퍼가 꽉 찼을 때 생산자가 대기해야 하고, 비었을 때 소비자가 대기해야 합니다. 공유 데이터(count, in/out 포인터)에 동시에 접근할 때 발생하는 **경쟁 상태(Race Condition)**를 막기 위해 세마포어나 뮤텍스 같은 동기화 도구가 필요합니다
