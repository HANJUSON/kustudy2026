트리 높이 비교 — 10억(10⁹) 키 저장 기준

트리의 높이 = 디스크 접근 횟수. RBT는 노드당 키 1개라 높이가 ~30까지 치솟지만, B-트리는 노드당 수백 키를 담아 높이 3으로 끝난다.

RBT — 디스크 접근 ~30회 ≈ 150ms B-트리 — 디스크 접근 3회

디스크 I/O 병목 — 메모리 대비 약 50,000배 느림

데이터가 주기억장치를 초과해 디스크에 저장되면 HDD는 메모리 대비 약 50,000배 느리다. 그래서 "접근 횟수(높이)"를 줄이는 것이 곧 성능이다.

디스크 / 메모리 = 50,000×

설계 목표: 노드 내부 탐색(키 수백 개 비교)은 빠른 CPU 연산으로 처리하고, 느린 디스크 접근 횟수 = 트리 높이를 최소화한다.

B-트리 노드 = 디스크 블록(약 4KB) — 한 번의 디스크 읽기로 수백 개 키를 적재

각 노드를 하나의 디스크 블록(약 4KB)에 대응시켜, 1회 디스크 읽기에서 수백 개의 키와 자식 포인터를 통째로 메모리로 가져온다. 레벨당 디스크 읽기 1회 → 높이 3이면 10억 키도 3회 읽기로 도달.

핵심 결론 — 왜 DB는 B-트리를 쓰는가

두 자료구조 모두 높이를 O(log n)으로 묶지만, 로그의 밑(base)이 다르다.

한 줄 요약: 분기 폭(차수)을 키워 높이를 낮추면, 가장 비싼 자원인 디스크 접근 횟수가 ~30회에서 3회로 약 10배 줄어든다. 이것이 B-트리가 DBMS·파일시스템의 표준 인덱스 구조인 이유다.
