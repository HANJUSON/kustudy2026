    3.학습한 알고리즘·운영체제 사례를 V 모델에 대입하여 요구사항 명세의 중요성을 체득한다.
# 학습내용
    1. V 모델(V-Model) 개요

    V 모델은 폭포수 모델을 발전시킨 SDLC 방법론으로, 개발 단계와 테스트 단계가 V자 대칭 구조로 대응된다.

    요구사항 분석 ↔ 인수 테스트(Acceptance Test)
    시스템 설계 ↔ 시스템 테스트(System Test)
    상세 설계 ↔ 통합 테스트(Integration Test)
    구현(코딩) ↔ 단위 테스트(Unit Test)
    핵심 원칙: 요구사항이 구체화되지 않으면 대응하는 테스트 기준도 정의할 수 없다.

    2. 요구사항 구체화가 필요한 사례 — 알고리즘 영역

    (1) 정렬 알고리즘 선택

    삽입 정렬: 거의 정렬된 소규모 데이터에서 O(n)에 근접 → 입력 분포 요구사항 필요
    퀵 정렬: 평균 O(n log n)이나 최악 O(n²) → 입력 분포·최악 허용 여부 명세 필요
    병합 정렬: 안정(stable) 정렬 보장 → 안정성 요구사항이 있을 때만 선택
    요구사항에 입력 크기·분포·안정성 필요 여부가 없으면 알고리즘 선택 자체가 불가능
    (2) DP(동적 프로그래밍) 설계

    최적 부분 구조(Optimal Substructure)와 중복 부분 문제(Overlapping Subproblems)를 요구사항으로 명확히 정의해야 구현 가능
    상태(state) 정의, 전이(transition) 함수, 기저 조건(base case)이 요구사항에 포함되어야 단위 테스트 작성 가능
    LCS, 행렬 연쇄 곱셈 등 → 점화식 자체가 요구사항이며 이를 기반으로 테스트 설계
    (3) 완전탐색·탐색 알고리즘

    DFS/BFS 적용 시 탐색 공간 범위(N의 크기)가 요구사항으로 정의되어야 함
    N이 불명확하면 O(N!) 완전탐색 허용 여부, 가지치기(pruning) 필요 여부 판단 불가
    탐색 알고리즘 선택 기준(깊이 우선 vs 너비 우선)도 요구사항(최단 경로 필요 여부 등)에 따라 결정
    3. 요구사항 구체화가 필요한 사례 — 운영체제 동기화 영역

    (1) Race Condition과 임계구역 요구사항

    count=5에서 producer/consumer 동시 실행 시 결과가 4 또는 6이 되는 비결정적 문제
    요구사항에 공유 자원 목록, 동시 접근 가능 프로세스 수, 임계구역 3조건(상호배제·진행·한계대기) 보장 여부가 없으면 설계 자체가 불완전
    선점형(Preemptive) vs 비선점형(Nonpreemptive) 커널 선택도 요구사항으로 명세 필요
    (2) Peterson's Solution과 하드웨어 환경 요구사항

    turn·flag 변수로 소프트웨어 해결책을 제시하지만, 현대 CPU/컴파일러의 명령어 재정렬로 인해 실제로는 동작 보장 불가
    Memory Barrier 삽입 필요 여부는 CPU 아키텍처·메모리 모델이 요구사항으로 명세되어야만 결정 가능
    하드웨어 요구사항(test_and_set, CAS) 없이는 올바른 동기화 솔루션 선택 불가
    (3) Mutex / Semaphore / Monitor 선택 기준

    Mutex Lock: 단순 상호배제·이진 락에 적합, Spinlock 사용 시 CPU 낭비 가능 → 대기 시간 요구사항 필요
    Counting Semaphore: 자원 N개 관리, 생산자-소비자 문제에 적합 → 초기값·wait/signal 호출 순서를 요구사항으로 명세
    Binary Semaphore: mutex lock과 유사하게 동작
    Monitor: 조건 변수(x_wait, x_signal)를 통한 복잡한 조건 동기화 → Signal and Wait vs Signal and Continue 중 어떤 방식인지 요구사항에 명세 필요
    도구 선택 기준이 요구사항에 없으면 구현 후 통합 테스트에서 Deadlock·Starvation 발생
    (4) Liveness 요구사항

    Deadlock: 두 프로세스가 서로의 자원을 무한 대기 → 요구사항에 Deadlock-free 보장 여부 명세
    Priority Inversion: 낮은 우선순위 프로세스가 semaphore 점유 시 발생 → 1997년 화성 탐사선 Pathfinder 실제 사례
    Priority Inheritance Protocol 적용 여부를 요구사항 단계에서 결정해야 설계 가능
    (5) 동기화 고전 문제 3가지와 요구사항 명세

    Bounded-Buffer: 버퍼 크기 N 명시, mutex는 반드시 counting semaphore wait 이후 호출 순서 준수 → 요구사항 미명세 시 Deadlock 발생
    Readers-Writers: readers 우선(First Problem) vs writers 우선(Second Problem) 정책을 요구사항으로 결정하지 않으면 Starvation 발생 가능
    Dining-Philosophers: deadlock-free와 starvation-free를 동시에 보장해야 하는지 요구사항 명세 필요 → 단순 semaphore 해결책은 deadlock 발생 가능
    4. V 모델 적용 방안 — 요구사항 체크리스트

    알고리즘: 입력 크기(N), 시간/공간 복잡도 허용 범위, 안정성 필요 여부 → 성능 테스트·경계값 테스트로 검증
    DP: 상태 정의, 전이 함수, 기저 조건 → 각 서브 문제 단위 테스트로 검증
    동기화: 공유 자원 목록, 임계구역 3조건, 동기화 도구 선택 기준, 우선순위 정책 → 동시성 테스트·Deadlock 검출로 검증
    OS 레벨: CPU 아키텍처, 메모리 모델, 커널 유형(선점/비선점) → 시스템 테스트로 검증
    5. 결론

    요구사항이 구체화되지 않으면 정렬 알고리즘 선택, DP 설계, 동기화 도구 결정 모두 올바른 결과를 보장할 수 없다. Race Condition·Deadlock·Priority Inversion은 요구사항 단계에서 공유 자원·접근 순서·우선순위 정책이 명세되지 않을 때 발생하는 설계 결함이다. V 모델은 각 개발 단계마다 테스트 기준을 미리 정의하여 구현 이후 오류 수정 비용을 줄이는 구조적 해답을 제공한다.
# 상세활동 내용
    1. V 모델(V-Model) 개요 학습

    소프트웨어 개발 방법론 중 하나인 V 모델의 구조를 학습하였다. V 모델은 폭포수 모델을 발전시킨 형태로, 개발 단계와 테스트 단계가 V자 형태로 대칭 대응되는 것이 핵심이다. 요구사항 분석은 인수 테스트, 시스템 설계는 시스템 테스트, 상세 설계는 통합 테스트, 구현은 단위 테스트와 각각 대응된다. 이를 통해 각 개발 단계에서 테스트 기준을 미리 정의하지 않으면 구현 이후 오류 발견이 늦어지고 수정 비용이 기하급수적으로 증가함을 이해하였다.

    2. 알고리즘 영역에서 요구사항 구체화 필요성 도출

    (1) 정렬 알고리즘 선택과 요구사항

    삽입·힙·병합·퀵 정렬의 시간복잡도와 특성을 비교하는 과정에서, 어떤 알고리즘이 최선인지는 요구사항이 구체적으로 정의되어야만 결정할 수 있다는 것을 확인하였다. 예를 들어 안정성(stable)이 보장되어야 하는 상황에서 퀵 정렬을 선택하면 잘못된 결과가 나올 수 있으며, 입력 데이터의 크기와 분포가 요구사항에 명시되어 있지 않으면 최악의 경우 O(n²)이 발생하는 알고리즘을 선택하는 실수를 범할 수 있다. V 모델 관점에서 요구사항 분석 단계에 입력 크기, 분포, 안정성 필요 여부를 반드시 명세해야 한다는 결론을 도출하였다.

    (2) DP 설계와 요구사항

    동적 프로그래밍(DP) 문제 해결 과정에서 상태(state) 정의, 전이(transition) 함수, 기저 조건(base case)이 명확하지 않으면 메모이제이션·타뷸레이션 구현 자체가 불가능함을 확인하였다. LCS, 행렬 연쇄 곱셈 등의 문제에서 점화식이 곧 요구사항이며, 이를 기반으로 각 서브 문제의 단위 테스트를 설계할 수 있다는 것을 V 모델과 연결지어 이해하였다.

    (3) 완전탐색·탐색 알고리즘과 요구사항

    DFS/BFS 탐색 알고리즘 적용 시 탐색 공간의 범위(N의 크기)가 요구사항으로 명확히 정의되어야 함을 학습하였다. N이 불명확하면 O(N!) 완전탐색이 허용되는지, 가지치기(Pruning)가 필요한지 판단할 수 없으며, 최단 경로 보장이 필요한지에 따라 DFS와 BFS 중 어떤 것을 선택할지도 달라지므로 이 역시 요구사항에 반드시 포함되어야 한다는 점을 확인하였다.

    3. 운영체제 동기화 영역에서 요구사항 구체화 필요성 도출

    (1) Race Condition과 임계구역 요구사항

    count=5인 상태에서 producer가 count++, consumer가 count--를 동시에 실행하면 결과가 4 또는 6이 되는 비결정적 문제를 통해 Race Condition의 본질을 이해하였다. 공유 자원의 목록, 동시 접근 가능한 프로세스 수, 임계구역의 상호배제·진행·한계대기 3조건 보장 여부가 요구사항에 명세되어 있지 않으면 설계 단계에서 올바른 해결책을 도출할 수 없음을 확인하였다. 또한 선점형(Preemptive)과 비선점형(Nonpreemptive) 커널의 특성 차이도 요구사항으로 명세되어야 한다는 점을 학습하였다.

    (2) Peterson's Solution과 하드웨어 환경 요구사항

    turn과 flag 변수를 이용한 소프트웨어 해결책인 Peterson's Solution을 분석하는 과정에서, 현대 CPU와 컴파일러의 명령어 재정렬로 인해 Memory Barrier 없이는 실제 동작을 보장할 수 없다는 한계를 확인하였다. 이를 통해 하드웨어 아키텍처와 메모리 모델이 요구사항에 명세되지 않으면 소프트웨어 레벨의 동기화 해결책만으로는 정확성을 보장할 수 없음을 이해하였다. test_and_set, CAS(compare_and_swap), Atomic Variables와 같은 하드웨어 기반 해결책의 선택 역시 하드웨어 요구사항이 전제되어야 결정 가능함을 확인하였다.

    (3) Mutex / Semaphore / Monitor 선택 기준과 요구사항

    세 가지 동기화 도구의 특성과 적합한 상황을 비교 분석하였다. Mutex Lock은 단순 상호배제에 적합하지만 Spinlock 사용 시 CPU 낭비가 발생할 수 있어 대기 시간 요구사항이 필요하다. Counting Semaphore는 자원 N개 관리와 생산자-소비자 문제에 적합하나 초기값과 wait/signal 호출 순서가 요구사항으로 명세되어야 한다. Monitor는 조건 변수(x_wait, x_signal)를 통한 복잡한 조건 동기화에 유리하지만 Signal and Wait 방식과 Signal and Continue 방식 중 어떤 것을 사용할지도 요구사항에 포함되어야 함을 확인하였다. 동기화 도구 선택 기준이 요구사항에 없으면 구현 후 통합 테스트 단계에서 Deadlock이나 Starvation이 발생할 수 있다는 점을 학습하였다.

    (4) Liveness 문제와 요구사항

    Deadlock과 Priority Inversion 문제를 학습하였다. Deadlock은 두 프로세스가 서로의 자원을 무한 대기하는 상태로, 요구사항에 Deadlock-free 보장 여부가 명세되어야 설계 단계에서 예방 전략을 수립할 수 있다. Priority Inversion은 1997년 화성 탐사선 Pathfinder에서 실제로 발생한 사례로, 낮은 우선순위 프로세스가 semaphore를 점유함으로써 높은 우선순위 프로세스가 블로킹되는 문제이다. Priority Inheritance Protocol 적용 여부를 요구사항 단계에서 결정해야만 올바른 설계가 가능하다는 점을 이해하였다.

    (5) 동기화 고전 문제 3가지 분석

    Bounded-Buffer, Readers-Writers, Dining-Philosophers 문제를 통해 요구사항 명세의 중요성을 구체적으로 확인하였다. Bounded-Buffer 문제에서는 버퍼 크기 N과 mutex는 반드시 counting semaphore wait 이후에 호출해야 한다는 순서 조건이 요구사항으로 명세되어야 Deadlock을 방지할 수 있다. Readers-Writers 문제에서는 readers 우선(First Problem)과 writers 우선(Second Problem) 중 어떤 정책을 선택할지가 요구사항으로 결정되지 않으면 Starvation이 발생할 수 있다. Dining-Philosophers 문제에서는 deadlock-free와 starvation-free를 동시에 보장해야 하는지를 요구사항에 명세하지 않으면 단순 semaphore 해결책이 deadlock을 유발할 수 있음을 확인하였다.

    4. V 모델 적용 방안 도출 및 정리

    학습한 내용을 종합하여 V 모델의 각 단계에서 구체화해야 할 요구사항 항목을 체크리스트로 정리하였다. 알고리즘 영역에서는 입력 크기·복잡도 허용 범위·안정성 필요 여부를, 동기화 영역에서는 공유 자원 목록·임계구역 3조건·동기화 도구 선택 기준·우선순위 정책을 요구사항 분석 단계에서 반드시 명세해야 함을 확인하였다. 또한 각 요구사항 항목이 V 모델의 어떤 테스트 단계(단위·통합·시스템·인수)와 대응되는지를 연결하여 이해함으로써, 요구사항 구체화가 단순한 문서 작성이 아니라 테스트 가능성(Testability)을 확보하기 위한 핵심 활동임을 체득하였다.
